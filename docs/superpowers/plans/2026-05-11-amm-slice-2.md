# AMM Slice 2 — Pool + seed + buy/sell + 0.3% fee + signed audit

> **For agentic workers:** REQUIRED SUB-SKILL: Use superpowers:subagent-driven-development. Steps use checkbox (`- [ ]`) syntax. Slice 2 of [RPOW Pool design spec](../specs/2026-05-11-amm-design.md).

**Goal:** Stand up the singleton constant-product AMM pool: seed it from admin balance, expose pool state, accept buy + sell swaps with 0.3% LP fee, sign every swap for audit. Pure DB — no Solana.

**Architecture:** Migration adds 4 AMM tables (pool, swaps, lp_balances, lp_events — `lp_events` is the audit table used by slice 3, but the schema lives here for one-shot migration). New module `apps/server/src/amm/math.ts` holds the pure-bigint math primitives (`isqrt`, `computeSwapOutput`, `computePriceImpactBps`). Extends `signing.ts` with `SwapPayload` + `signSwapPayload`. Five new endpoints in `apps/server/src/routes/amm/`. Every state-changing op: `withTx` → `SELECT FOR UPDATE` on `amm_pool` → validate → mutate → INSERT signed audit row → assert invariant.

**Tech Stack:** Postgres 17, Fastify 4, zod, vitest, node:crypto (ed25519).

## Conventions

- All money math in `BigInt`. Never `Number`.
- All state-changing ops open `withTx`, then `SELECT … FOR UPDATE` on the pool row first thing. The seed endpoint additionally validates that no pool row exists yet.
- Every swap inserts an `amm_swaps` row with a fresh ed25519 signature over the canonical `SwapPayload`. Same pattern as `FlipPayload` / `MatchPayload`.
- After every state mutation, assert `new_R_rpow × new_R_usdc ≥ old_R_rpow × old_R_usdc` (the invariant grows monotonically with fees). Throw if it ever decreases — that means a math bug.
- Body validation via zod. Stringified BigInts as input (`amount_base_units: "1000000"`), regex `^[1-9][0-9]*$`. Output strings same.
- Every endpoint checks `isAllowed(app.config.ammAllowedEmails, email)` AND `readTermsAcceptedAt(...)` is non-null. Returns `403 NOT_ALLOWED` or `403 TERMS_NOT_ACCEPTED`. (Public read endpoints — `/amm/pool`, `/amm/swaps/recent` — skip both checks. Quote endpoints require allowlist but not terms.)

The invariant: total USDC held by the server (across user balances + pool reserve) must never exceed `AMM_USDC_POOL_CAP_BASE_UNITS`. Only operations that INCREASE the total need the check — admin/credit (this slice) and the slice-5 deposit indexer (later). Swap/LP ops don't change total USDC.

```ts
// Inside admin.ts credit handler, before the UPDATE:
const totalRes = await app.pool.query<{ total: string }>(
  `SELECT
     (
       (SELECT COALESCE(SUM(usdc_base_units), 0) FROM users)
       + COALESCE((SELECT usdc_reserve_base_units FROM amm_pool WHERE id='main'), 0)
     )::text AS total`,
);
const currentTotal = BigInt(totalRes.rows[0].total);
const cap = BigInt(app.config.ammUsdcPoolCapBaseUnits);
if (currentTotal + amount > cap) {
  return reply.code(409).send({
    error: 'USDC_POOL_CAP_EXCEEDED',
    message: `system-wide USDC cap is ${cap} base units; current ${currentTotal}, would add ${amount}`,
  });
}
```

(The COALESCE for amm_pool handles the pre-seed state when no pool row exists yet.)

```ts
// Pure-bigint math for the constant-product AMM. No DB, no IO.

/** Integer square root of a non-negative bigint (Newton's method). */
export function isqrt(n: bigint): bigint {
  if (n < 0n) throw new Error('isqrt: negative input');
  if (n < 2n) return n;
  let x = n;
  let y = (x + 1n) / 2n;
  while (y < x) {
    x = y;
    y = (x + n / x) / 2n;
  }
  return x; // floor(sqrt(n))
}

export interface SwapInputs {
  reserveIn: bigint;
  reserveOut: bigint;
  amountIn: bigint;
  /** Fee numerator (default 997 = 0.3% fee). */
  feeNum?: bigint;
  /** Fee denominator (default 1000). */
  feeDen?: bigint;
}

/** Uniswap V2 constant-product swap output:
 *  amountOut = (reserveOut × amountIn × feeNum) / (reserveIn × feeDen + amountIn × feeNum)
 *  The fee stays in the pool — caller adds the FULL amountIn to reserveIn after this.
 *  Returns 0n for non-positive amountIn (caller should reject). */
export function computeSwapOutput({ reserveIn, reserveOut, amountIn, feeNum = 997n, feeDen = 1000n }: SwapInputs): bigint {
  if (amountIn <= 0n) return 0n;
  if (reserveIn <= 0n || reserveOut <= 0n) throw new Error('computeSwapOutput: reserves must be positive');
  const inWithFee = amountIn * feeNum;
  const numerator = inWithFee * reserveOut;
  const denominator = reserveIn * feeDen + inWithFee;
  return numerator / denominator; // floor
}

/** Fee paid (in `amountIn` units) for a swap. Returns `amountIn × (feeDen - feeNum) / feeDen`. */
export function computeFeeIn(amountIn: bigint, feeNum: bigint = 997n, feeDen: bigint = 1000n): bigint {
  if (amountIn <= 0n) return 0n;
  return (amountIn * (feeDen - feeNum)) / feeDen;
}

/** Effective price impact in basis points, comparing spot vs effective output.
 *  Returns a non-negative bigint (rounded down). */
export function computePriceImpactBps(reserveIn: bigint, reserveOut: bigint, amountIn: bigint, amountOut: bigint): bigint {
  if (amountIn <= 0n || amountOut <= 0n) return 0n;
  const spotNumerator = reserveOut * amountIn;
  const effectiveNumerator = amountOut * reserveIn;
  if (spotNumerator <= effectiveNumerator) return 0n;
  return ((spotNumerator - effectiveNumerator) * 10_000n) / spotNumerator;
}

/** Reserve product (k = R_in × R_out). Used for invariant assertions. */
export function poolK(rpow: bigint, usdc: bigint): bigint {
  return rpow * usdc;
}
```

Tests cover:
- `isqrt(0) = 0`, `isqrt(1) = 1`, `isqrt(4) = 2`, `isqrt(10) = 3` (floor), `isqrt(10n**18n)` correctness.
- `isqrt(negative)` throws.
- `computeSwapOutput`: spot-price scenario with no fee (feeNum=feeDen=1n) returns exact y = R_out × x / (R_in + x). With 0.3% fee, output is strictly less than the no-fee output. Output is 0 for amountIn=0.
- Invariant property: `(R_in + amountIn) × (R_out - amountOut) ≥ R_in × R_out` for all positive amountIn (the new k strictly grows because of the fee).
- `computeFeeIn`: `amountIn=1000` → fee=3.
- `computePriceImpactBps`: known scenario (huge swap relative to pool moves the impact close to 10000 bps; small swap → small bps).

```ts
export interface SwapPayload {
  id: string;
  account_email_hash: string;
  direction: 'BUY' | 'SELL';
  rpow_delta_base_units: bigint;
  usdc_delta_base_units: bigint;
  fee_base_units: bigint;
  pool_rpow_after: bigint;
  pool_usdc_after: bigint;
  created_at: string;
}

function canonicalSwap(payload: SwapPayload): Buffer {
  const ordered = JSON.stringify(
    {
      id: payload.id,
      account_email_hash: payload.account_email_hash,
      direction: payload.direction,
      rpow_delta_base_units: payload.rpow_delta_base_units,
      usdc_delta_base_units: payload.usdc_delta_base_units,
      fee_base_units: payload.fee_base_units,
      pool_rpow_after: payload.pool_rpow_after,
      pool_usdc_after: payload.pool_usdc_after,
      created_at: payload.created_at,
    },
    (_, v) => (typeof v === 'bigint' ? v.toString() : v),
  );
  return Buffer.from(ordered, 'utf8');
}

export function signSwapPayload(payload: SwapPayload, privHex: string): Buffer {
  return sign(null, canonicalSwap(payload), privKeyFromHex(privHex));
}

export function verifySwapPayload(payload: SwapPayload, sig: Buffer, pubHex: string): boolean {
  return verify(null, canonicalSwap(payload), pubKeyFromHex(pubHex), sig);
}
```

Tests:
- sign + verify roundtrip
- tampered payload rejected
- deterministic output (Ed25519)
- BigInt encoding (values serialize as strings)

### `POST /amm/seed { rpow_base_units, usdc_base_units }` (admin only)

Behavior:
1. Auth check (401 UNAUTHORIZED).
2. Admin check via `isAmmAdmin` (403 NOT_ADMIN).
3. Body parse: both fields stringified-bigint, positive (400 BAD_REQUEST).
4. `withTx`:
   - `SELECT FOR UPDATE` on `amm_pool` — confirm zero rows. If a row already exists, return 409 POOL_ALREADY_SEEDED.
   - Verify caller's USDC balance ≥ usdc input. Verify caller's RPOW balance via `burnFromUser` flow (throws INSUFFICIENT_BALANCE).
   - Debit `users.usdc_base_units -= usdc_input` for caller.
   - `burnFromUser(c, caller, rpow_input, signingKey)` — same pattern as gladiator/trivia seeding.
   - Decrement `minted_supply` by rpow_input (matching the burn-supply convention).
   - Compute `initialLp = isqrt(rpow_input * usdc_input)`. Require `initialLp > MIN_LIQUIDITY` (1000) — else 400 INVALID_AMOUNT.
   - INSERT `amm_pool { id='main', rpow_reserve_base_units, usdc_reserve_base_units, total_lp_supply: initialLp, fee_bps: 30 }`.
   - INSERT `amm_lp_balances { account_email: caller, lp_balance: initialLp - MIN_LIQUIDITY }`. The MIN_LIQUIDITY units stay in `total_lp_supply` but credited to no one.
5. Return `{ initial_lp: (initialLp - MIN_LIQUIDITY), total_lp: initialLp, pool: {…} }`.

Tests (admin-allowlisted caller assumed unless otherwise stated):
- 401 without session
- 403 NOT_ADMIN for non-admin
- 400 BAD_REQUEST for missing/invalid body
- 400 INSUFFICIENT_USDC if caller doesn't have enough USDC
- 409 INSUFFICIENT_BALANCE if caller doesn't have enough RPOW (via burnFromUser throw)
- 200 happy path: pool row + admin LP balance match the math; reserves = inputs; total_lp = isqrt(in_a × in_b)
- 409 POOL_ALREADY_SEEDED on second call

Both endpoints share this skeleton:

```ts
async function executeSwap(req, reply, app, direction: 'BUY' | 'SELL') {
  // 1. Auth + allowlist + terms.
  // 2. Body parse: amount_in, min_out.
  // 3. withTx:
  //    a. SELECT FOR UPDATE amm_pool WHERE id='main'.
  //    b. If no row: 503 POOL_NOT_SEEDED.
  //    c. Compute output via computeSwapOutput.
  //    d. If output < min_out: throw SLIPPAGE_EXCEEDED (→ 400).
  //    e. Direction-specific debit/credit:
  //       BUY (USDC in, RPOW out):
  //         - users.usdc_base_units -= amount_in (insufficient → INSUFFICIENT_USDC).
  //         - Mint RPOW token of value=output to caller, signed via signTokenPayload, cap-check minted_supply.
  //         - Pool: R_usdc += amount_in, R_rpow -= output.
  //       SELL (RPOW in, USDC out):
  //         - burnFromUser(c, caller, amount_in, signingKey).
  //         - users.usdc_base_units += output.
  //         - Pool: R_rpow += amount_in (full), R_usdc -= output.
  //         - minted_supply -= amount_in (burn) AND minted_supply += output? No — the
  //           USDC side doesn't touch minted_supply (USDC isn't RPOW). RPOW side:
  //           burned amount_in stays burned (minted_supply -= amount_in); the pool's
  //           R_rpow grew but it's NOT credited to anyone's token balance, so no
  //           minted_supply offset. Net minted_supply -= amount_in.
  //    f. UPDATE amm_pool SET rpow_reserve_base_units=..., usdc_reserve_base_units=....
  //    g. fee_paid_in = amount_in * 3 / 1000 (in the input asset's base units).
  //    h. INSERT amm_swaps with signed SwapPayload.
  //    i. Assert invariant: new_R_rpow * new_R_usdc >= old_R_rpow * old_R_usdc. Throw if violated.
  // 4. Reply.
}
```

A subtle accounting point on `minted_supply` for SELL: the user's RPOW is burned (minted_supply -= amount_in), and the pool's R_rpow grows by amount_in (but R_rpow is NOT in `tokens` table — it's just an accounting number on `amm_pool`). So no offsetting mint. The total RPOW "in circulation" effectively shrinks by amount_in on a SELL (it's in the pool, not in any user's token list). Same as gladiator session bankroll. Document in the route comment.

For BUY (mint): the pool's R_rpow shrinks (just an accounting decrement), and we mint a real `tokens` row to the user. `minted_supply` must grow by `output` (the new token's value) AND we ALSO need to grow `minted_supply` to reflect that pool-rpow is leaving "pool accounting" and entering "user balance accounting". Hmm — actually `minted_supply` tracks total tokens in circulation. If pool R_rpow is NOT in minted_supply (it shouldn't be — those RPOW are escrowed in the pool, not minted), then when we mint `output` to the user, `minted_supply += output` (cap-checked).

This convention is the SAME as gladiator: bankroll RPOW are debited from minted_supply when burned into the session; payout RPOW are minted (+= minted_supply) on win.

So:
- BUY: pool R_rpow decreases, user gets minted RPOW (+= minted_supply, cap-check).
- SELL: pool R_rpow increases (from a burned RPOW), minted_supply -= amount_in (the burn).

This matches the gladiator convention. Good.

Tests:
- 401 no session
- 403 NOT_ALLOWED (not in AMM_ALLOWED_EMAILS)
- 403 TERMS_NOT_ACCEPTED
- 503 POOL_NOT_SEEDED (no pool row)
- BUY happy path: USDC debited, RPOW credited (tokens row exists), reserves updated, signature in amm_swaps row verifies.
- SELL happy path: RPOW burned, USDC credited, reserves updated, signed.
- 400 SLIPPAGE_EXCEEDED: min_out too high.
- 400 INSUFFICIENT_USDC: BUY with insufficient USDC balance.
- 409 INSUFFICIENT_BALANCE: SELL with insufficient RPOW.
- 503 SUPPLY_CAP_REACHED: BUY large enough to bust minted_supply cap (corner case).
- Invariant holds after every swap (assert in test: re-fetch reserves, compute k, compare to pre).
